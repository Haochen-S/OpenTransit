# Email Verification Challenge

OpenTransit Sydney uses a lightweight server-side verification challenge before sending passwordless email login codes.

This document explains the design rationale, security properties, limitations, and future upgrade path for the verification flow.

## Overview

The application uses passwordless email OTP login. Before a user can request a one-time login code, they must complete a short verification challenge generated by the backend.

Supported challenge types include:

* **Math challenge** — for example, `What is 7 + 4?`
* **Text challenge** — for example, `Type these characters: A7K2M`

The challenge is designed to reduce casual automated abuse of the email sending endpoint while keeping the sign-in flow simple and mobile-friendly.

## How It Works

The verification flow is handled by the backend:

1. The frontend requests a challenge from `GET /api/auth/captcha`.
2. The backend generates a random `captcha_id` and challenge.
3. The expected answer is stored server-side as a hash, not plaintext.
4. The user submits the answer when requesting an email login code.
5. The backend validates the answer before sending the OTP email.
6. The challenge expires after a short TTL and is single-use.

This verification layer is used together with:

* Per-IP rate limits on authentication endpoints
* Per-email limits for login code requests
* Hashed email OTP codes
* OTP expiry times
* Maximum verification attempts per OTP

## Design Rationale

### Lightweight User Experience

The application is designed for quick mobile use. A short text-based challenge avoids heavy third-party widgets, image captchas, and multi-step verification flows.

This keeps the login process simple while still adding friction against basic automated requests.

### Lower Operational Complexity

Using an internal verification challenge avoids the setup and maintenance overhead of third-party captcha providers during early deployment.

There are no additional captcha API keys, dashboards, billing rules, or external client scripts required for the first public release.

### Reduced Email Abuse

Email OTP login depends on a send-code endpoint. Without protection, automated scripts could repeatedly trigger login emails and create unnecessary SMTP load or user annoyance.

The verification challenge adds a pre-send check before the email system is used.

### Practical Security Trade-off

This design is not intended to replace enterprise-grade bot detection. It is a practical trade-off for a small public transport application with limited traffic and a low-risk account model.

The goal is to reduce casual abuse while keeping the system easy to operate, inexpensive to run, and simple for real users.

## Security Properties

| Control                | Purpose                                                |
| ---------------------- | ------------------------------------------------------ |
| Server-side generation | Prevents the frontend from deciding the correct answer |
| Hashed answers         | Avoids storing challenge answers in plaintext          |
| Short TTL              | Limits the useful lifetime of a challenge              |
| Single-use challenges  | Prevents replay after successful validation            |
| Rate limiting          | Reduces automated request volume                       |
| Email send limits      | Prevents repeated OTP emails to the same address       |
| OTP expiry             | Limits the lifetime of login codes                     |
| OTP attempt limits     | Reduces brute-force attempts against codes             |

## Limitations

This approach has several known limitations:

| Limitation                   | Explanation                                                                                 |
| ---------------------------- | ------------------------------------------------------------------------------------------- |
| Weak against advanced bots   | Simple math or text challenges can be solved by scripted clients                            |
| No risk scoring              | The system does not classify suspicious IPs, VPNs, or datacenter traffic                    |
| No vendor abuse intelligence | It does not benefit from provider-managed bot detection models                              |
| In-memory challenge storage  | If deployed across multiple backend instances, shared storage such as Redis would be needed |
| Limited visibility           | There is no external dashboard for captcha analytics or abuse monitoring                    |

The challenge should be treated as one layer of defence, not a complete bot protection system.

## When This Approach Is Appropriate

This approach is suitable when:

* The application is in early public release
* Traffic is limited or moderate
* The main risk is casual abuse of email sending
* A low-friction mobile login experience is important
* The project should avoid unnecessary third-party dependencies and operating cost

## When To Upgrade

A third-party captcha or bot protection service should be considered if:

* Login email abuse increases
* The application receives significant public traffic
* The backend is scaled to multiple instances
* SMTP cost, deliverability, or abuse reporting becomes a concern
* Stronger bot detection is required

Possible upgrade options include:

* Cloudflare Turnstile
* reCAPTCHA
* hCaptcha
* A shared Redis-backed challenge store combined with stronger request monitoring

The passwordless OTP flow can remain the same. Only the pre-send verification step would need to be replaced or strengthened.

## Summary

OpenTransit Sydney uses a lightweight server-side verification challenge as a practical balance between usability, security, performance, and operating cost.

It helps reduce basic automated email abuse without adding third-party captcha complexity. For larger-scale production use, this layer should be upgraded to a stronger bot protection system.
